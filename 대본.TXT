01. 오늘 구성
오늘 강의의 목표는 대표님이 개발자를 평가하는 사람이 되는 것이 아닙니다.
개발자와 같은 지도를 보면서 질문하고, 문서 기준으로 의사결정할 수 있게 되는 것입니다.
오늘은 네 부분으로 갑니다. 1부에서는 서비스가 실제로 어떻게 움직이는지 보고, 2부에서는 채용 면접에서 무엇을 물어야 하는지 봅니다.
3부에서는 외주를 맡길 때 계약서와 요구사항 문서에 무엇을 남겨야 하는지 정리하고, 마지막 4부에서는 Codex 앱을 설치해서 프로젝트 폴더와 연결하는 흐름까지 봅니다.

02. 강사 소개
짧게 제 소개를 드리겠습니다. 저는 게더링 대표이사 김대일입니다.
비전공자 출신으로 개발과 창업을 같이 해왔고, 외주 개발도 20건 이상 수행했습니다.
전국 대학 동아리 앱 마이캠퍼스를 운영하면서 서비스 기획, 개발, 운영, 외주와 채용 문제를 직접 겪었습니다.
오늘 내용은 개발 문법 강의가 아니라, 대표가 개발자와 대화할 때 놓치면 안 되는 기준을 정리하는 시간이라고 보시면 됩니다.

03. 서비스 흐름
서비스는 화면 하나로 보이지만 안쪽에는 프런트, API, 백엔드, DB, 서버, Git, 배포가 연결돼 있습니다.
자동차로 비유하면 프런트는 운전자가 보는 계기판과 외장, API는 부품끼리 신호를 주고받는 배선과 커넥터, 백엔드는 엔진과 제어장치, DB는 정비 기록부와 블랙박스, 서버는 차가 실제로 달리는 도로와 차고, Git은 설계도와 정비 이력, 배포는 차를 고객에게 출고하는 과정에 가깝습니다.
프런트는 고객이 보는 화면, API는 화면과 처리부의 약속, 백엔드는 규칙 처리, DB는 기록, 서버는 실행 공간, Git은 코드 원본, 배포는 고객에게 공개하는 절차입니다.
대표가 이 구조를 알아야 하는 이유는 직접 개발하기 위해서가 아닙니다.
"이 기능은 화면만 바꾸는 일인가, 저장과 권한과 배포까지 건드리는 일인가"를 구분하기 위해서입니다.
앞으로 각 장은 세 가지로 보시면 됩니다. 첫째, 이게 무슨 기능인지. 둘째, 서비스 안에서 어떤 역할을 하는지. 셋째, 나중에 개발할 때 무엇을 미리 정해야 하는지입니다.
미니맵은 지금 우리가 어느 부분을 말하고 있는지 계속 확인하는 기준으로 보면 됩니다.

04. 프런트엔드 선택
프런트엔드는 고객과 대표가 직접 보는 화면입니다.
자동차로 보면 계기판, 핸들, 버튼, 외장처럼 사람이 직접 보고 만지는 부분입니다.
무슨 기능이냐면 화면을 보여주고, 고객 입력을 받고, 버튼 클릭을 API로 보내고, 결과를 다시 화면에 보여주는 부분입니다.
역할은 고객 경험을 책임지는 것입니다. 같은 기능이라도 로딩 표시, 오류 문구, 성공 후 이동이 다르면 고객은 전혀 다르게 느낍니다.
HTML과 CSS는 화면의 뼈대이고, JavaScript와 React, Vue, Angular는 화면이 바뀌고 움직이는 방식을 담당합니다.
개발 전에 대표가 정해야 할 것은 필요한 화면, 버튼, 입력칸, 성공 후 이동, 실패 문구입니다.
그래서 대표는 "최신 기술인가요?"보다 "다음 개발자가 이어받기 쉬운 화면 규칙이 있나요?"를 물어야 합니다.

05. API
API는 프런트와 백엔드가 주고받는 주문서입니다.
자동차로 보면 계기판의 버튼과 엔진 제어장치 사이를 이어주는 배선, 커넥터, 신호 규칙이라고 보면 됩니다.
무슨 기능이냐면 화면에서 보낸 요청을 백엔드가 이해할 수 있는 약속으로 만드는 것입니다.
역할은 누가, 어떤 값을, 어떤 권한으로 요청했는지 전달하고, 성공과 실패 결과를 화면에 돌려주는 것입니다.
그래서 API는 "무슨 값을 보내고, 누가 요청할 수 있고, 성공하면 무엇이 오고, 실패하면 무엇이 오는지"를 정하는 문서가 있어야 합니다.
예약 신청 API라면 보내는 값은 날짜, 시간, 고객 ID이고 성공 응답은 예약번호입니다.
실패 응답에는 마감, 권한 없음, 중복 신청 같은 이유가 들어가야 합니다.
여기서 API 키는 비밀번호라기보다 출입증에 가깝습니다. 같은 예약 요청이라도 우리 서비스가 보낸 요청인지 확인하게 해주는 값입니다.
개발 전에 대표는 보내는 값, 권한, API 키, 성공 응답, 실패 응답, 고객에게 보일 문구가 문서에 있는지 확인해야 합니다.

06. 백엔드 선택
백엔드는 화면 뒤에서 권한을 확인하고, 사업 규칙을 처리하고, DB에 저장하고, API 응답을 돌려주는 부분입니다.
자동차로 보면 엔진과 제어장치입니다. 운전자가 페달을 밟으면 실제로 힘을 만들고, 속도와 안전 조건을 계산하는 쪽입니다.
무슨 기능이냐면 화면 뒤에서 실제 판단과 처리를 하는 부분입니다.
역할은 "이 고객이 이 행동을 해도 되는가", "결제나 예약 규칙에 맞는가", "실패했다면 어떤 이유인가"를 결정하는 것입니다.
Python은 빠른 API나 AI 연동, 내부 도구에 잘 맞고, Node는 JavaScript 또는 TypeScript로 가벼운 API와 실시간 처리에 자주 쓰입니다.
Java와 Spring Boot는 결제, 정산, 권한, 기업 내부 시스템처럼 안정성과 규칙이 중요한 곳에서 많이 씁니다.
하지만 대표가 봐야 할 것은 언어 이름보다 우리 사업의 규칙과 실패 처리를 어디서 담당하는가입니다.
개발 전에는 회원, 예약, 결제, 환불, 정산, 권한처럼 자동 처리할 규칙과 예외 케이스를 먼저 정해야 합니다.
"권한, 결제, DB 저장, 실패 기록을 어디서 어떻게 처리하나요?"라고 물어보면 됩니다.

07. DB
DB는 개발 용어가 아니라 회사의 장부입니다.
자동차로 보면 차량 등록증, 정비 이력, 주행 기록, 블랙박스 같은 기록 저장소입니다.
무슨 기능이냐면 고객, 주문, 결제, 상담, 로그 같은 정보를 저장하고 다시 조회하게 해주는 것입니다.
역할은 회사 장부처럼 기록을 남기고, 누가 볼 수 있는지 제한하고, 문제가 생겼을 때 복구할 기준을 갖는 것입니다.
고객, 주문, 결제, 로그가 무엇으로 남는지 정해야 하고, 누가 조회하고 수정하고 삭제할 수 있는지도 나눠야 합니다.
여기에 하나 더 붙이면 "언제 지우고 언제 남길 것인가"입니다.
고객 탈퇴 시 지울 정보와 결제, 정산처럼 보관해야 할 정보를 구분하지 않으면 운영 중 문제가 됩니다.
개발 전에는 어떤 정보를 저장할지, 어떤 정보는 삭제할지, 백업은 언제 하고 어디까지 복구할지 정해야 합니다.
대표는 DB를 만들 줄 몰라도 "무엇이 남고, 누가 볼 수 있고, 언제 복구할 수 있는가"는 반드시 물어야 합니다.

08. 서버
서버는 서비스가 실제로 돌아가는 운영 환경입니다.
자동차로 보면 차가 실제로 달리는 도로, 주차장, 정비소 같은 운영 공간입니다.
무슨 기능이냐면 고객이 접속했을 때 우리 서비스가 실행되는 공간을 제공하는 것입니다.
역할은 접속 처리, 백업, 장애 알림, 운영 비용과 접근 권한의 책임 지점이 되는 것입니다.
기술 선택지로는 AWS, Azure, Naver Cloud 같은 클라우드와 자체 서버가 있습니다.
클라우드를 쓰느냐 자체 서버를 쓰느냐는 기술 취향이 아니라 책임 구조의 문제입니다.
초기 서비스는 보통 클라우드가 유리하지만, 회사 명의 계정인지, 월 비용 상한이 있는지, 백업과 장애 알림이 있는지 확인해야 합니다.
개발 전에는 서버 계정 소유자, 월 비용 상한, 백업 위치, 장애 알림 수신자, 외주사의 접근 권한을 정해야 합니다.
외주사가 만든 서버가 외주사 계정에만 있으면 나중에 장애나 이전 때 통제권이 흔들립니다.

09. Git
Git은 코드 원본과 변경 이력을 남기는 회사 자산입니다.
자동차로 보면 설계도와 정비 이력부입니다. 언제 어떤 부품을 왜 바꿨는지 남아 있어야 다음 정비사가 이어받을 수 있습니다.
무슨 기능이냐면 코드 원본을 보관하고, 누가 언제 무엇을 고쳤는지 남기는 것입니다.
역할은 되돌리기, 인수인계, 협업의 기준이 되는 것입니다.
대표가 코드를 몰라도 저장소 소유자가 회사 계정인지, 누가 언제 무엇을 고쳤는지 남는지는 확인해야 합니다.
GitHub나 GitLab은 단순한 개발 도구가 아니라 다음 개발자에게 코드를 넘겨주는 인수인계 장부입니다.
개인 계정에 코드가 있으면 납품 후 유지보수와 이전이 막힐 수 있습니다.
개발 전에는 저장소 소유자, 접근 권한, 운영 코드와 개발 코드의 분리, 변경 이유를 남기는 기준을 정해야 합니다.
외주 종료 전에는 회사 계정에서 저장소, 접근 권한, 배포 이력을 확인할 수 있어야 합니다.

10. 배포
배포는 만든 기능을 고객에게 실제로 공개하는 절차입니다.
자동차로 보면 공장에서 만든 차를 검사하고, 시운전하고, 고객에게 출고하는 과정입니다.
무슨 기능이냐면 개발된 기능을 테스트 환경을 거쳐 운영 환경에 반영하는 것입니다.
역할은 "언제 고객에게 열 것인가", "누가 확인하고 승인할 것인가", "문제가 생기면 어떻게 되돌릴 것인가"를 정하는 것입니다.
운영 서버에 바로 올리는 것이 아니라 테스트 서버에서 대표가 먼저 확인하고, 필요하면 베타 테스트로 일부 사용자에게 먼저 열어볼 수 있습니다.
대표는 어떤 기준을 통과해야 운영 서버로 올리는지, 문제가 생기면 언제 직전 버전으로 되돌릴지 정해야 합니다.
개발 전에는 테스트 서버, 베타 운영, 운영 반영 기준, 오픈 후 점검 항목, 되돌리기 기준을 정해야 합니다.
배포는 개발자의 작업 완료가 아니라 테스트, 베타 확인, 운영 전환, 복구 기준까지 포함한 운영 절차로 봐야 합니다.

11. 버튼 하나의 흐름
작아 보이는 버튼 하나에도 전체 흐름이 붙습니다.
자동차로 보면 시동 버튼 하나를 누르는 것과 비슷합니다. 버튼은 작지만 배터리, 엔진, 센서, 안전 조건, 계기판 표시가 함께 움직입니다.
무슨 기능이냐면 고객 행동 하나가 화면, API, 백엔드, DB, 다시 화면으로 이어지는 전체 동작입니다.
역할은 기능 범위를 현실적으로 보여주는 것입니다. 버튼 하나가 단순 문구 수정인지, 저장과 권한과 알림까지 포함하는지 구분하게 해줍니다.
고객이 버튼을 누르면 API 요청이 가고, 백엔드가 권한과 중복을 검사하고, DB에 저장하고, 다시 화면에 결과가 돌아옵니다.
그래서 버튼이 바꾸는 화면, 데이터, 알림, 관리자 기록까지 한 번에 물어봐야 합니다.
개발 전에는 이 버튼이 보내는 값, 권한, 중복 클릭, 실패 문구, DB 저장, 관리자 화면 반영 여부를 정해야 합니다.
버튼 하나라도 DB 저장, 권한, 알림, 관리자 화면까지 붙으면 별도 기능 범위로 보는 것이 맞습니다.

12. 1부 정리
앞 내용을 다 잊어도 이것만 기억하면 됩니다.
기능은 화면 하나가 아니라 고객 행동, 화면 상태, API 요청, 백엔드 규칙, DB 기록, 운영 확인이 이어진 흐름입니다.
자동차로 말하면 외장만 보는 것이 아니라 계기판, 배선, 엔진, 정비 기록, 도로, 설계도, 출고 절차를 함께 보는 것입니다.
예를 들어 예약 버튼이면 고객이 누르고, 화면은 처리 중을 보여주고, API가 요청을 보내고, 백엔드는 권한과 중복을 검사하고, DB에 기록한 뒤 성공 또는 실패 화면으로 돌려줍니다.
대표는 각 기술의 이름을 외우는 것보다 "무슨 기능인지, 어떤 역할인지, 개발 전에 무엇을 정해야 하는지"를 물어보면 됩니다.
CRUD, API, DB, Git, 배포, 운영이라는 단어만 잡아도 개발자와 대화할 때 범위를 훨씬 명확하게 잡을 수 있습니다.

13. 채용 핵심
채용에서는 프로젝트 이름보다 기능 경험을 봐야 합니다.
말을 잘하는 후보가 아니라 실제로 만들 수 있는 후보를 찾는 것이 목표입니다.
검증 순서는 프로젝트, 역할, 기능, 예외, 운영입니다.
후보가 어디까지 직접 했고, 실패했을 때 무엇을 바꿨는지 들어야 합니다.
이제부터는 우리 서비스에 필요한 기능을 면접 질문으로 바꾸는 방법을 보겠습니다.

14. 기능 경험 매칭
우리 서비스의 핵심 기능과 후보의 경험을 직접 맞춰 봅니다.
결제가 중요한 서비스라면 결제 실패, 환불, 중복 클릭을 설명할 수 있어야 합니다.
예약 변경이 중요하다면 시간 변경, 알림, 취소 기준을 말할 수 있어야 합니다.
관리자가 중요한 서비스라면 권한표와 차단 화면을 설명할 수 있어야 합니다.
우리 서비스에 없는 멋진 경험보다, 실제로 필요한 기능 경험을 먼저 보는 것이 중요합니다.

15. 면접 질문 프레임
후보가 "해봤습니다"라고 말하면 바로 믿지 말고 다섯 단계로 쪼개서 물어봅니다.
누가 쓰는 서비스였는지, 본인이 맡은 범위는 어디였는지, 직접 만든 기능은 무엇인지 묻습니다.
그 다음 실패와 권한은 어떻게 처리했는지, 마지막으로 배포 후 문제를 본 적이 있는지 확인합니다.
좋은 답은 본인 역할과 실패 처리, 운영 확인까지 이어집니다.

16. 좋은 답 나쁜 답
나쁜 답은 주어가 흐립니다. "팀에서 했습니다", "연동했습니다", "오류는 다른 분이 봤습니다"처럼 말합니다.
좋은 답은 본인 역할이 분명합니다. "저는 실패 응답 처리와 재시도 화면을 맡았습니다"처럼 말합니다.
대표는 주어가 팀이면 한 번 더 물어보고, 주어가 "저는"으로 바뀌는지 확인해야 합니다.
그리고 답변 안에 화면, 파일, 문서, 확인 방법이 들어오는지 보면 됩니다.

17. 신입 주니어 판단
신입과 주니어에게 완벽함을 기대하면 안 됩니다.
대신 면접에서 작은 상황 질문을 던졌을 때, 어떤 화면과 데이터와 예외를 먼저 확인하는지 봐야 합니다.
예를 들어 "관리자 목록에 상태 필터가 필요했다면 무엇부터 확인하겠습니까?"라고 물어볼 수 있습니다.
기능명만 말하는 후보보다 순서와 예외를 말하는 후보가 더 구체적입니다.
실제 일을 시키지 않고도 사고 과정을 말로 확인할 수 있습니다.

18. 포트폴리오 질문 검증
포트폴리오는 예쁜 화면보다 후보가 설명하는 역할과 결정 과정을 봐야 합니다.
이 화면에서 본인이 맡은 부분, 어려웠던 예외, 나중에 고친 이유를 질문합니다.
결과물은 있는데 본인 판단을 설명하지 못하면 보류해야 합니다.
꼬리질문은 "그때 직접 바꾼 화면, 파일, 문서는 무엇인가요?"입니다.

19. 2부 정리
면접 전에 세 가지를 정합니다.
첫째, 우리 핵심 기능 3개입니다.
둘째, 각 기능에서 실패 케이스가 무엇인지입니다.
셋째, 본인 역할과 예외 처리와 운영 확인을 합격 기준으로 둡니다.
면접 중에는 좋은 말이 아니라 증거와 본인 역할을 기록해야 합니다.

20. 외주 핵심
외주는 직원을 뽑는 것이 아니라 제작을 맡기는 일입니다.
대표는 제작자가 아니라 기준을 정하는 사람입니다.
무엇을 만들고 무엇을 안 만들지, 검수 기준은 무엇인지, 소유권과 유지보수는 어떻게 할지 문서로 남겨야 합니다.
"알아서 잘 만들어 주세요"라는 말은 외주에서 가장 위험한 문장입니다.
이제부터는 외주 전에 대표가 정해야 할 기준을 하나씩 보겠습니다.

21. 역할 분리
대표가 정할 것과 외주사가 설명할 것을 분리해야 합니다.
대표는 목표, 범위, 우선순위를 정합니다.
외주사는 구현 방식, 일정, 비용을 설명합니다.
검수 기준과 변경 승인은 양쪽이 계약 전에 같이 확인합니다.
이 구분이 없으면 외주사가 추측해서 만들고, 검수 때 분쟁이 생깁니다.

22. MVP 범위
처음부터 완성품을 외주로 만들려고 하면 비용이 커집니다.
이번 버전에서 고객이 반드시 끝까지 해야 하는 한 가지 흐름을 먼저 정합니다.
신청, 결제, 문의 접수처럼 매출이나 문의와 직접 연결되는 흐름이 우선입니다.
고급 통계, 자동 추천, 복잡한 쿠폰은 뒤로 미루는 것이 좋습니다.
외주는 MVP 제작에 맞고, 운영과 고도화는 내부 기준을 세우면서 키우는 편이 안전합니다.

23. 요구사항 문서
요구사항은 기능명보다 흐름과 예외가 중요합니다.
"주문 결제"라고만 쓰면 부족합니다.
성공 시 어디로 이동하는지, 실패 시 어떤 문구를 보여주는지, 권한은 어떻게 막는지, 처리 중에는 버튼을 비활성화하는지 적어야 합니다.
문장 공식은 "사용자가 행동하면 시스템이 검사한 뒤 성공 결과와 실패 문구를 보여준다"는 형태입니다.
이렇게 써야 개발자와 대표가 같은 장면을 보고 검수할 수 있습니다.

24. 검수 기준
검수 기준은 납품 후가 아니라 계약 전에 적어야 합니다.
대표가 기획서 순서대로 직접 클릭하고, 성공뿐 아니라 실패, 취소, 권한 차단까지 실행합니다.
검수 캡처, 권한표, 관리자 화면, 납품 문서를 증거로 남깁니다.
성공만 동작하는 기능은 완료로 보지 않는다는 기준이 필요합니다.

25. 계약 범위
계약 범위에는 포함과 제외를 같이 써야 합니다.
포함 범위에는 화면 수, 관리자 화면 수, 수정 횟수, 배포 횟수를 씁니다.
제외 범위에는 디자인 전면 개편, 신규 기능, 앱스토어 심사, PG 계약, 유료 API 비용 같은 항목을 분리합니다.
요구사항이 바뀌면 변경 요청서, 추가 견적, 일정 영향을 확인한 뒤 승인해야 합니다.
포함만 있고 제외가 없으면 나중에 "당연히 해주는 줄 알았다"는 말이 나오기 쉽습니다.

26. 소유권과 계정
소스코드와 계정은 회사가 소유해야 합니다.
Git 저장소는 회사 계정이 소유하고 대표가 관리자 권한을 가져야 합니다.
클라우드와 서버도 회사 명의로 두고 결제 카드, 알림, 접근 권한을 확인합니다.
도메인과 DNS도 회사 명의로 두고, 외주사는 초대 권한만 사용하는 구조가 안전합니다.
납품물은 화면만이 아니라 소스코드, 계정, 배포 문서까지 포함된다고 생각해야 합니다.

27. 유지보수
오픈 이후의 비용과 대응 시간을 먼저 정합니다.
긴급 장애는 서비스 접속 불가나 결제 중단처럼 매출과 직접 연결되는 문제입니다.
이 경우 30분 내 접수, 4시간 내 1차 조치 같은 기준을 둘 수 있습니다.
일반 오류와 신규 기능은 별도로 구분하고, 문서에 없던 새 동작은 별도 견적으로 진행합니다.
유지보수 계약은 "계속 봐드립니다"가 아니라 월 비용, 포함 범위, 응답 시간으로 적어야 합니다.

28. 3부 정리
외주 발주 전 마지막 체크는 범위, 검수, 소유권, 유지보수, 장애입니다.
계약서에는 화면 수와 제외 범위, 실패와 권한을 포함한 검수 기준, 회사 계정 소유권, 월 유지보수 비용, 장애 대응 시간을 남겨야 합니다.
이 다섯 가지가 없으면 나중에 비용과 책임을 다시 협상하게 됩니다.
대표가 모든 기술을 알 필요는 없지만, 기준을 문서로 남기는 역할은 반드시 해야 합니다.

29. Codex 앱 소개
이제 마지막으로 Codex 앱 설치 흐름을 보겠습니다.
Codex 앱은 프로젝트 폴더를 읽고 같이 작업하는 AI 개발 에이전트라고 이해하면 됩니다.
브라우저에서 질문만 하는 것이 아니라, 내 컴퓨터의 프로젝트 폴더를 연결해 파일 구조를 읽고 수정 방향을 제안할 수 있습니다.
오늘 목표는 깊은 개발 작업이 아니라 설치, 로그인, 폴더 연결까지 한 번에 실습하는 것입니다.

30. 활용 예시
설치 후에는 처음부터 큰 작업을 시키기보다 작은 질문으로 시작하면 좋습니다.
"이 폴더가 어떤 서비스인지 설명해줘"라고 하면 폴더 구조를 읽고 프로젝트 성격을 요약하게 할 수 있습니다.
그 다음 "버튼 이름과 안내 문구를 더 쉽게 바꿔줘"처럼 작은 수정 요청을 할 수 있습니다.
오류가 나면 "오류 메시지의 원인을 쉬운 말로 정리해줘"라고 물어볼 수도 있습니다.
핵심은 Codex를 개발자 대신 모든 것을 맡기는 도구가 아니라, 프로젝트를 같이 읽고 확인하는 동료처럼 쓰는 것입니다.

31. 설치 전 필요 사항
설치 전에 ChatGPT 계정이 필요합니다.
Codex 앱 로그인에 ChatGPT 계정을 사용하므로, 먼저 이메일과 비밀번호로 브라우저에서 로그인되는지 확인합니다.
가능하면 Windows 업데이트도 확인해 두면 설치 과정에서 막힐 가능성이 줄어듭니다.
Git이 있으면 변경 이력을 확인하기 쉽지만, 오늘 설치 흐름에서는 필수라고 보지 않아도 됩니다.
먼저 계정 로그인이 되는지 확인하는 것이 가장 중요합니다.

32. 스토어 설치
Codex 앱은 Microsoft Store에서 설치합니다.
시작 메뉴에서 Microsoft Store를 열고, 검색창에 Codex 또는 OpenAI Codex를 입력합니다.
앱 이름과 로고를 확인한 뒤 다운로드하고 실행합니다.
설치가 끝나면 바로 로그인 화면으로 넘어갈 수 있습니다.
이때 비슷한 이름의 앱을 헷갈리지 않도록 게시자와 앱 이름을 확인하는 습관을 가지면 좋습니다.

33. ChatGPT 연동
Codex 앱을 처음 실행하면 ChatGPT 계정과 연동해야 합니다.
앱에서 Sign in with ChatGPT를 누르면 브라우저 로그인 창이 열릴 수 있습니다.
브라우저에서 계정 인증을 마치면 다시 Codex 앱으로 돌아옵니다.
이때 앱 화면에 워크스페이스나 프로젝트를 추가할 수 있는 화면이 보이는지 확인합니다.
로그인이 안 되면 먼저 브라우저에서 ChatGPT 로그인이 정상인지 다시 확인하면 됩니다.

34. 프로젝트 폴더 생성
Codex에 연결할 프로젝트 폴더를 먼저 만듭니다.
예를 들어 바탕화면에 homepage-renewal 같은 이름의 폴더를 만들 수 있습니다.
폴더 이름은 너무 일반적인 "새 폴더"보다 프로젝트 목적이 보이게 정하는 것이 좋습니다.
그 안에 기획서, 사진 자료, 로고 파일처럼 Codex가 참고할 자료를 넣어봅니다.
처음에는 작고 명확한 폴더 하나로 시작하는 것이 좋습니다.

35. Codex에 폴더 연결
이제 Codex 앱에서 Add project를 선택하고 바탕화면의 프로젝트 폴더를 연결합니다.
폴더를 연결한 뒤에는 "이 기획서와 사진으로 홈페이지 초안을 만들어줘"처럼 요청할 수 있습니다.
또는 "이 폴더가 어떤 프로젝트인지 설명해줘"처럼 먼저 읽어달라고 해도 됩니다.
중요한 것은 Codex가 볼 수 있는 자료가 폴더 안에 있어야 한다는 점입니다.
파일이 흩어져 있으면 답도 흐려지므로, 프로젝트별로 폴더를 나누는 습관을 권합니다.

36. 4부 정리
Codex 앱 설치 흐름은 네 단계로 기억하면 됩니다.
첫째, ChatGPT 계정을 준비합니다.
둘째, Microsoft Store에서 Codex 앱을 설치합니다.
셋째, ChatGPT 계정으로 로그인합니다.
넷째, 바탕화면의 프로젝트 폴더를 Codex에 연결합니다.
대표가 기억할 문장은 "회원가입을 먼저 끝내고, Microsoft Store에서 Codex 앱을 설치한 뒤, 내 프로젝트 폴더를 연결한다"입니다.

37. 마무리
오늘 가져갈 핵심은 여섯 가지입니다.
첫째, 프런트가 정보를 보내고 백엔드가 처리해 DB에 저장합니다.
둘째, 긴 설명보다 성공, 실패, 권한, 복구 기준을 물어야 합니다.
셋째, 채용은 프로젝트명이 아니라 기능 경험으로 검증합니다.
넷째, 외주는 제작이고 대표는 범위와 기준을 맡습니다.
다섯째, 소스코드, 서버, 계정, 유지보수는 계약서에 적어야 합니다.
마지막으로 Codex 앱은 Microsoft Store에서 설치하고 프로젝트 폴더와 연결해서 시작하면 됩니다.
오늘 끝나고 바로 할 일은 우리 핵심 기능 3개를 고르고, 각 기능마다 실패 케이스 1개와 외주 계약 문장 1개를 적어보는 것입니다.
