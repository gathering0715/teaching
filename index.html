1부 · 기초 / 소통

서비스 흐름은 화면·처리·저장·운영이 이어지는 구조다

대표가 봐야 할 것화면에서 누른 행동이 실제로는 규칙, 저장, 서버 운영까지 이어지는지 확인한다.
개발자에게 물을 것이 기능은 프런트만 바꾸나요, API·DB·배포까지 건드리나요?
대표가 확인할 것고객 행동, 성공/실패 화면, 저장 기록, 장애 책임자를 한 줄로 연결한다.
프런트: 고객 경험 API: 약속 백엔드: 규칙 DB: 기록 서버: 운영 Git: 코드 원본 배포: 반영 책임

프런트엔드

프런트엔드는 고객이 보고 누르는 화면이다

프런트가 하는 일 화면 표시 입력 받기 정보 전달 결과 보여주기
HTML / CSS장점: 단순함단점: 화면 상태가 많아지면 관리가 어렵다
React장점: 채용 쉬움단점: 초기에 규칙을 안 잡으면 복잡해진다
Vue장점: 적응 빠름단점: 팀 규칙이 없으면 화면마다 달라진다
바닐라 JS장점: 처음은 가벼움단점: 서비스가 커지면 수정 비용이 늘기 쉽다
Angular장점: 규칙 강함단점: 작은 팀에는 학습과 채용 부담이 있다
Svelte장점: 가볍고 빠름단점: 인력 풀이 React보다 좁다
1. 주 목적프런트가 하는 일: 화면 표시 · 입력 받기 · 정보 전달 · 결과 보여주기 2. 개발언어HTML · CSS · JavaScript · React · Vue · Angular · Svelte 3. 개발 전 생각할 것서비스에 어떤 화면이 필요한가 / 각 화면에 어떤 버튼과 기능이 들어가는가
서비스 구상고객이 왜 들어오고 무엇을 끝내야 하는지 적는다
페이지 구성각 페이지에 들어갈 문구, 버튼, 입력칸, 기능을 떠올린다
변경 대비나중에 자주 바뀔 화면과 기능을 미리 구분한다
대표는 "무엇이 최신인가"보다 "우리 회사가 1년 뒤에도 고칠 수 있는가"를 본다.

API

API는 화면과 처리부가 주고받는 약속이다

API가 하는 일 화면 정보 받기 필요한 정보 전달 처리 결과 돌려주기 오류 구분
REST Fetch GraphQL OpenAPI 문서
1. 주 목적화면에서 어떤 정보를 보내고 처리 뒤 무엇을 보여줄지 정한다
2. 연결 방식REST · Fetch · GraphQL · OpenAPI 문서
3. 개발 전 생각할 것버튼을 누르면 어떤 정보가 넘어가고 성공·실패 때 무엇을 보여줄 것인가
어떤 정보를 보낼지 누가 할 수 있는지 성공하면 무엇을 보여줄지 실패하면 무엇을 알려줄지 고객에게 보일 문구 문제 추적 기록
꼭 물을 것화면에서 보낼 정보와 보여줄 결과를 정했나
예시예약 신청: 성공·실패·권한 오류
체크 기준보내는 정보, 성공·실패 화면, 고객 표시 문구가 함께 정해졌는가
예약 신청 API 예시화면에서 보내는 정보: 날짜·시간·고객ID성공: 예약번호실패: 마감·권한 없음·중복 신청

백엔드

백엔드는 화면 뒤에서 규칙을 처리하는 영역이다

백엔드가 하는 일 권한 확인 규칙 처리 DB 저장 처리 결과 전달
1. 주 목적화면 뒤에서 권한, 규칙, 저장, 처리 결과를 만든다
2. 개발언어Python · Node · Java · PHP · Spring
3. 개발 전 생각할 것회원, 예약, 결제, 정산 같은 사업 규칙을 어디까지 자동 처리할 것인가
Python / FastAPI

장점: 요즘 트렌드단점: 큰 서비스는 구조를 잡아야 한다

Python / Django

장점: 완성형 웹단점: 자유도가 낮게 느껴질 수 있다

Node / Express

장점: 가벼움단점: 팀 규칙이 없으면 코드가 흩어진다

Node / NestJS

장점: 팀 개발 구조단점: 초기 학습량이 있다

Java / Spring Boot

장점: 기업용 표준단점: MVP에는 무겁게 느껴질 수 있다

PHP / Laravel

장점: 웹 제작 빠름단점: 요즘 채용이 거의 불가능할 수 있다

서비스 규칙누가 무엇을 할 수 있고 어떤 조건에서 막을지 정한다
자동 처리예약 확정, 결제 확인, 알림 발송 같은 흐름을 떠올린다
운영 기록문제가 생겼을 때 확인해야 할 기록을 미리 생각한다
기술 선택 3문장: 우리 기능의 규칙을 처리할 수 있는가 / 데이터 저장을 안전하게 하는가 / 장애 때 원인을 추적할 수 있는가.

DB

DB는 서비스 정보를 저장하는 회사 장부다

DB가 하는 일 데이터 저장 조회 권한 제한 복구 기준
PostgreSQL MySQL MongoDB Redis
1. 주 목적고객, 주문, 결제, 로그를 저장하고 다시 조회한다
2. 저장소 종류PostgreSQL · MySQL · MongoDB · Redis
3. 개발 전 생각할 것고객, 주문, 결제, 상담, 로그 중 어떤 정보를 DB에 저장할 것인가

저장

고객, 주문, 결제, 로그가 무엇으로 남는지 정한다.

복구

백업 주기, 보관 위치, 복구 시간, 책임자를 정한다.

권한

누가 조회·수정·삭제하고 원문을 볼 수 있는지 나눈다.

체크 기준관리자는 주문을 보되 결제 정보 원문까지 볼 필요가 있는가?
체크 기준백업 주기와 장애 시 복구 목표 시간이 정해져 있는가?
대표 질문고객 탈퇴, 결제 취소, 정산 완료 후 각각 어떤 기록을 남기나요?
고객 탈퇴 시 삭제할 정보와 결제·정산 기록처럼 보관할 정보를 구분한다.

서버

서버는 서비스가 실제로 돌아가는 운영 공간이다

서버가 하는 일 서비스 실행 접속 처리 백업 장애 알림
AWS Azure Naver Cloud 자체 서버
1. 주 목적서비스가 실제로 돌아가는 운영 환경을 제공한다
2. 운영 환경AWS · Azure · Naver Cloud · 자체 서버
3. 개발 전 생각할 것서비스가 열린 뒤 비용, 백업, 장애 알림, 계정 권한을 누가 볼 것인가
클라우드초기 서비스 기본값초기 비용이 낮고 확장·백업·장애 알림 도구를 쓰기 쉽다. 대신 권한 설정과 월 비용 관리를 확인해야 한다.
자체 서버실전담 인력 있을 때물리 통제는 가능하지만 장비·공간·전기·보안·장애 출동을 내부에서 감당해야 한다.
비용월 비용 상한을 정한다
장애알림과 복구 책임자를 정한다
계약 확인장애 알림·백업·접근 권한을 적는다
체크 기준운영 서버 계정, 백업, 장애 알림, 접근 권한을 회사가 확인할 수 있는가?
계약 전 확인: 서버 계정 소유자, 월 비용 상한, 백업 위치, 장애 알림 수신자.

Git

Git은 코드 원본과 변경 이력을 보관하는 도구다

Git이 하는 일 변경 이력 원본 보관 되돌리기 협업 기준
GitHub GitLab Branch Commit
Branch운영 중인 코드와 개발 중인 코드를 분리한다
Commit변경 이유와 수정 범위를 짧게 남긴다
Review운영 반영 전 최소 확인 절차를 둔다
저장소 소유권회사 계정이 원본 저장소를 소유하는가
변경 이력누가 언제 무엇을 고쳤는지 남는가
인수인계다음 개발자가 코드를 받을 수 있는가
체크 기준운영 반영 기준 브랜치와 변경 이유가 남아 있는가?
대표 질문외주 종료 후 회사 계정에서 저장소, 권한, 배포 이력을 모두 확인할 수 있나요?

배포

배포는 만든 기능을 고객에게 공개하는 절차다

배포가 하는 일 테스트 반영 운영 공개 오픈 확인 되돌리기
테스트 서버 베타 테스트 운영 서버 점검
검수 환경운영 전 테스트 서버에서 대표가 확인하는가
베타 운영일부 고객이나 내부 사용자로 먼저 확인하는가
오픈 후 점검운영 반영 뒤 핵심 기능과 오류를 바로 확인하는가

동작 흐름

버튼 하나는 화면·처리·저장을 함께 움직인다

버튼 하나에서 확인할 일 고객 행동 화면 상태 서버 판단 기록과 응답
누가 눌렀나 무슨 값을 보냈나 성공 기준은 뭔가 어디에 기록되나
예: 예약 신청 버튼 고객 클릭 → 중복 예약 차단 → DB 저장 → 관리자 알림 → 성공·실패 화면까지가 한 기능 범위다.
프런트 클릭 정보 전달 백엔드 검사 DB 저장 결과 돌려주기 화면 반영
권한로그인·역할·상태를 먼저 확인하는가
중복두 번 눌러도 한 번만 저장되는가
실패실패 문구와 재시도 방법이 보이는가
기록관리자 화면과 로그에서 추적 가능한가
대표 질문이 버튼의 범위는 화면 변경만인가요, DB 저장까지인가요?
문서 문장예약 신청 버튼은 권한 확인, 중복 방지, DB 저장, 성공·실패 화면 반영까지 포함한다.
버튼 하나라도 DB 저장, 권한, 알림, 관리자 화면까지 붙으면 별도 기능 범위로 본다.

2부 · 채용

채용은 우리 서비스에 필요한 기능부터 정리한다

면접 장면 "채팅 기능 해봤습니다"

우리 서비스에 필요한 기능과 연결되지 않으면 판단 기준이 흔들린다.

1단계우리 서비스에 필요한 기능 정리
2단계그 기능을 확인할 면접 질문 정리
3단계좋은 답변을 찾고 이끌어낸다

1. 필요한 기능 정리

우리 서비스에 필요한 기능 경험부터 정리한다

우리 핵심예약 변경상태값 · 알림 · 취소 기준
후보 A채팅 UI매칭 약함
후보 B관리자 권한매칭 강함
후보 C알림 누락 처리매칭 강함
대표 판단먼저 우리 서비스 핵심 기능을 적어야, 후보 경험이 필요한 경험인지 판단할 수 있다.

2. 면접 질문 정리

필요한 기능을 면접 질문으로 바꾼다

01무슨 서비스?목적 · 사용자
02본인 역할?화면 · API · DB
03직접 만든 기능?입력 · 처리 · 저장
04실패하면?권한 · 중복 · 오류
05운영에서?로그 · 배포 · 복구
대표 판단후보가 말한 프로젝트를 우리에게 필요한 기능 질문으로 좁혀서 본다.
후보 답"결제 연동 해봤습니다."
추가 질문실패 화면, 중복 결제, 환불 기준, 운영 기록까지 물어본다
추가 질문 필요: 성공했다는 말만 반복함 실제 경험 신호: 실패·중복·기록까지 설명함

3. 좋은 답변 찾고 이끌기

좋은 답변은 역할과 근거가 보인다

흐린 답"팀에서 결제 연동했습니다."본인 역할이 안 보임
좋은 답"저는 실패 화면과 재시도 안내를 맡았습니다."맡은 범위가 보임
대표 판단 기준고객에게 보이는 실패 화면까지 설명하면 실제 구현 경험에 가깝다.

3. 좋은 답변 찾고 이끌기

작은 상황 질문을 하면 답변이 구체해진다

면접 질문 예시관리자 목록에 상태 필터가 필요했다면?

어떤 화면 · 데이터 · 예외를 먼저 확인할지 묻는다

신입에게 볼 것순서를 말하는가
주니어에게 볼 것예외를 떠올리는가
위험 신호질문을 기능명으로만 답함
대표 판단실제 일을 시키지 않고도, 후보가 기능을 어떤 순서로 생각하는지 말로 확인한다.

4부 · Codex 설치

Codex 앱은 내 컴퓨터에서 함께 일하는 개발 보조 도구다

무엇인가 AI 개발 에이전트 프로젝트 폴더를 읽고 수정 방향을 제안한다
어디서 쓰나 Windows 앱 설치 후 앱 화면에서 프로젝트별로 대화한다
무엇을 하나 읽기 · 수정 · 확인 코드 설명, 작은 수정, 테스트 실행을 도와준다
오늘 목표Codex 앱이 무엇인지 이해하고, 설치 후 ChatGPT 계정과 연결해 프로젝트 폴더를 여는 상태까지 간다.

Step 1

설치 후에는 이런 일부터 시켜볼 수 있다

이해프로젝트 구조 설명이 폴더가 어떤 서비스인지 쉽게 설명해달라고 묻는다
개선문구·화면 수정작은 UI 문구, 버튼 이름, 안내 문장을 바꿔본다
검증오류 원인 찾기에러 메시지와 실행 결과를 보고 원인을 정리한다
대표 멘트"처음부터 큰 개발을 맡기기보다 설명, 작은 수정, 오류 확인 순서로 익숙해지면 됩니다."

Step 2

설치 전에 ChatGPT 계정이 필요하다

필수ChatGPT 회원가입Codex 앱 첫 실행 때 ChatGPT 계정으로 로그인한다
권장Windows 최신 업데이트Windows 11이 가장 안정적이고, Windows 10은 최신 상태가 좋다
있으면 좋음Git 설치프로젝트 변경 이력을 확인하고 되돌리기 쉬워진다
강조ChatGPT 회원가입과 로그인은 필수로 설명한다. 앱만 설치해도 계정 연결 전에는 제대로 시작할 수 없다.

Step 3

구글에서 Codex app을 검색한다

01Google 열기브라우저 주소창이나 검색창 사용
02codex app 검색영문으로 검색하면 공식 결과를 찾기 쉽다
03OpenAI 공식 결과 선택developers.openai.com 또는 openai.com 경로 확인
04광고 피하기비공식 다운로드 파일은 누르지 않는다
검색어codex app 또는 openai codex app windows로 검색한다.

Step 4

공식 페이지에서 Windows 앱을 설치한다

01Windows 앱 버튼공식 페이지에서 Windows 설치 버튼을 누른다
02Microsoft StoreStore 화면으로 넘어가면 설치를 누른다
03Codex 열기설치 후 Open 또는 시작 메뉴에서 실행한다
대체 경로스토어가 바로 열리지 않으면 PowerShell에서 winget install Codex -s msstore를 사용할 수 있다.

Step 5

Codex 앱을 ChatGPT 계정과 연동한다

로그인Sign in with ChatGPT브라우저 로그인 창이 열리면 ChatGPT 계정으로 인증한다
복귀Codex 앱으로 돌아오기인증이 끝나면 앱에서 계정과 워크스페이스를 확인한다
ChatGPT 회원가입 브라우저 인증 앱으로 복귀

Step 6

먼저 바탕화면에 프로젝트 폴더를 만든다

1바탕화면 열기바탕화면으로 이동
2새 폴더 만들기예: My Codex Project
3파일 넣기홈페이지, 문서, 코드 파일을 넣어둔다
4이름 정리프로젝트 목적이 보이게 폴더명을 정한다
폴더 예시바탕화면\homepage-renewal처럼 바탕화면에 한 프로젝트당 한 폴더를 만든다.

Step 7

Codex 앱에서 프로젝트 폴더를 추가한다

1Add projectCodex 앱에서 프로젝트 추가를 누른다
2폴더 선택방금 바탕화면에 만든 프로젝트 폴더를 선택한다
3첫 질문"이 폴더에 무엇이 있는지 설명해줘"
4작은 요청문구 수정, 파일 정리 같은 작업부터 시작한다
안전 기준처음에는 설명 요청으로 시작하고, 큰 변경은 어떤 파일을 바꿀지 먼저 물어본다.

대표가 가져갈 6가지

1프런트가 정보를 보내고, 백엔드가 처리해 DB에 저장한다.
2긴 설명보다 성공, 실패, 권한, 복구 기준을 묻는다.
3채용은 프로젝트명이 아니라 기능 경험으로 검증한다.
4외주는 제작이고, 대표는 범위와 기준을 맡는다.
5소스코드, 서버, 계정, 유지보수는 계약서에 적는다.
6Codex 앱은 공식 사이트에서 Windows 설치 화면으로 들어가 받는다.
