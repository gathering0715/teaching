1부 · 기초 / 소통

서비스 흐름은 화면·처리·저장·운영이 이어지는 구조다

보는 기준: 기능 · 역할 · 개발 전 고려 프런트: 화면과 입력 API: 요청과 응답 약속 백엔드: 규칙 처리 DB: 기록과 복구 서버: 실행 공간 Git: 코드 원본 배포: 고객 공개

프런트엔드

프런트엔드는 고객이 보고 누르는 화면이다

프런트의 기능 / 역할 화면 표시 입력 받기 API 결과 보여주기
HTML / CSS장점: 단순함단점: 화면 상태가 많아지면 관리가 어렵다
React장점: 채용 쉬움단점: 초기에 규칙을 안 잡으면 복잡해진다
Vue장점: 적응 빠름단점: 팀 규칙이 없으면 화면마다 달라진다
바닐라 JS장점: 처음은 가벼움단점: 서비스가 커지면 수정 비용이 늘기 쉽다
Angular장점: 규칙 강함단점: 작은 팀에는 학습과 채용 부담이 있다
Svelte장점: 가볍고 빠름단점: 인력 풀이 React보다 좁다
1. 무슨 기능인가고객이 보는 화면, 입력칸, 버튼, 이동 흐름을 만든다 2. 어떤 역할인가고객 행동을 받아 API로 보내고 결과를 다시 보여준다 3. 개발 전 고려필요한 화면, 버튼, 오류 문구, 나중에 자주 바뀔 영역을 정한다
서비스 구상고객이 왜 들어오고 무엇을 끝내야 하는지 적는다
페이지 구성각 페이지에 들어갈 문구, 버튼, 입력칸, 기능을 떠올린다
변경 대비나중에 자주 바뀔 화면과 기능을 미리 구분한다
대표는 "무엇이 최신인가"보다 "우리 회사가 1년 뒤에도 고칠 수 있는가"를 본다.

API

API는 화면과 처리부가 주고받는 약속이다

API의 기능 / 역할 화면 정보 받기 필요한 정보 전달 처리 결과 돌려주기 오류 구분
REST Fetch GraphQL OpenAPI 문서
1. 무슨 기능인가화면과 백엔드가 주고받을 요청·응답 형식을 정한다
2. 어떤 역할인가누가 무엇을 요청했고 성공·실패 결과가 무엇인지 구분한다
3. 개발 전 고려보낼 값, 권한, API 키, 성공·실패 응답을 문서로 남긴다
어떤 정보를 보낼지 누가 할 수 있는지 성공하면 무엇을 보여줄지 실패하면 무엇을 알려줄지 고객에게 보일 문구 문제 추적 기록
꼭 물을 것화면에서 보낼 정보와 보여줄 결과를 정했나
API 키우리 서비스가 보낸 요청인지 확인하는 출입증
체크 기준보내는 정보, 성공·실패 화면, 고객 표시 문구가 함께 정해졌는가
예약 신청 API 예시화면에서 보내는 정보: 날짜·시간·고객IDAPI 키: 우리 서비스가 보낸 요청인지 확인하는 출입증성공: 예약번호실패: 마감·권한 없음·중복 신청

백엔드

백엔드는 화면 뒤에서 규칙을 처리하는 영역이다

백엔드의 기능 / 역할 권한 확인 규칙 처리 DB 저장 처리 결과 전달
1. 무슨 기능인가화면 뒤에서 권한, 검증, 계산, 저장 요청을 처리한다
2. 어떤 역할인가사업 규칙을 코드로 실행하고 실패 이유를 남긴다
3. 개발 전 고려회원, 예약, 결제, 정산 규칙과 예외 케이스를 먼저 정한다
Python / FastAPI

장점: 요즘 트렌드단점: 큰 서비스는 구조를 잡아야 한다

Python / Django

장점: 완성형 웹단점: 자유도가 낮게 느껴질 수 있다

Node / Express

장점: 가벼움단점: 팀 규칙이 없으면 코드가 흩어진다

Node / NestJS

장점: 팀 개발 구조단점: 초기 학습량이 있다

Java / Spring Boot

장점: 기업용 표준단점: MVP에는 무겁게 느껴질 수 있다

PHP / Laravel

장점: 웹 제작 빠름단점: 요즘 채용이 거의 불가능할 수 있다

서비스 규칙누가 무엇을 할 수 있고 어떤 조건에서 막을지 정한다
자동 처리예약 확정, 결제 확인, 알림 발송 같은 흐름을 떠올린다
운영 기록문제가 생겼을 때 확인해야 할 기록을 미리 생각한다
기술 선택 3문장: 우리 기능의 규칙을 처리할 수 있는가 / 데이터 저장을 안전하게 하는가 / 장애 때 원인을 추적할 수 있는가.

DB

DB는 서비스 정보를 저장하는 회사 장부다

DB의 기능 / 역할 데이터 저장 조회 권한 제한 복구 기준
PostgreSQL MySQL MongoDB Redis
1. 무슨 기능인가고객, 주문, 결제, 로그를 저장하고 다시 조회한다
2. 어떤 역할인가회사의 장부처럼 기록, 권한, 복구 기준을 책임진다
3. 개발 전 고려무엇을 저장·삭제·보관하고 누가 볼 수 있는지 정한다

저장

고객, 주문, 결제, 로그가 무엇으로 남는지 정한다.

복구

백업 주기, 보관 위치, 복구 시간, 책임자를 정한다.

권한

누가 조회·수정·삭제하고 원문을 볼 수 있는지 나눈다.

체크 기준관리자는 주문을 보되 결제 정보 원문까지 볼 필요가 있는가?
체크 기준백업 주기와 장애 시 복구 목표 시간이 정해져 있는가?
대표 질문고객 탈퇴, 결제 취소, 정산 완료 후 각각 어떤 기록을 남기나요?
고객 탈퇴 시 삭제할 정보와 결제·정산 기록처럼 보관할 정보를 구분한다.

서버

서버는 서비스가 실제로 돌아가는 운영 공간이다

서버의 기능 / 역할 서비스 실행 접속 처리 백업 장애 알림
AWS Azure Naver Cloud 자체 서버
1. 무슨 기능인가서비스가 실제로 실행되고 접속되는 공간을 제공한다
2. 어떤 역할인가운영 비용, 백업, 장애 알림, 접근 권한의 책임 지점이다
3. 개발 전 고려회사 계정, 비용 상한, 백업, 알림 수신자를 정한다
클라우드초기 서비스 기본값초기 비용이 낮고 확장·백업·장애 알림 도구를 쓰기 쉽다. 대신 권한 설정과 월 비용 관리를 확인해야 한다.
자체 서버실전담 인력 있을 때물리 통제는 가능하지만 장비·공간·전기·보안·장애 출동을 내부에서 감당해야 한다.
비용월 비용 상한을 정한다
장애알림과 복구 책임자를 정한다
계약 확인장애 알림·백업·접근 권한을 적는다
체크 기준운영 서버 계정, 백업, 장애 알림, 접근 권한을 회사가 확인할 수 있는가?
계약 전 확인: 서버 계정 소유자, 월 비용 상한, 백업 위치, 장애 알림 수신자.

Git

Git은 코드 원본과 변경 이력을 보관하는 도구다

Git의 기능 / 역할 변경 이력 원본 보관 되돌리기 협업 기준
GitHub GitLab Branch Commit
1. 무슨 기능인가코드 원본과 누가 무엇을 바꿨는지 남긴다
2. 어떤 역할인가되돌리기, 인수인계, 협업의 기준이 된다
3. 개발 전 고려회사 계정 소유, 접근 권한, 변경 이력 기준을 정한다
저장소 소유권회사 계정이 원본 저장소를 소유하는가
변경 이력누가 언제 무엇을 고쳤는지 남는가
인수인계다음 개발자가 코드를 받을 수 있는가
체크 기준운영 반영 기준 브랜치와 변경 이유가 남아 있는가?
대표 질문외주 종료 후 회사 계정에서 저장소, 권한, 배포 이력을 모두 확인할 수 있나요?

배포

배포는 만든 기능을 고객에게 공개하는 절차다

배포의 기능 / 역할 테스트 반영 운영 공개 오픈 확인 되돌리기
테스트 서버 베타 테스트 운영 서버 점검
무슨 기능인가완성된 기능을 고객이 쓰는 운영 환경에 반영한다
어떤 역할인가테스트, 승인, 공개, 되돌리기 절차를 책임진다
개발 전 고려운영 반영 기준과 문제 발생 시 복구 방법을 정한다

동작 흐름

버튼 하나는 화면·처리·저장을 함께 움직인다

기능 하나: 역할 / 개발 전 고려 고객 행동 화면 상태 서버 판단 기록과 응답
누가 눌렀나 무슨 값을 보냈나 성공 기준은 뭔가 어디에 기록되나
예: 예약 신청 버튼 고객 클릭 → 중복 예약 차단 → DB 저장 → 관리자 알림 → 성공·실패 화면까지가 한 기능 범위다.
프런트 클릭 API 백엔드 검사 DB 저장 결과 돌려주기 화면 반영
권한로그인·역할·상태를 먼저 확인하는가
중복두 번 눌러도 한 번만 저장되는가
실패실패 문구와 재시도 방법이 보이는가
기록관리자 화면과 로그에서 추적 가능한가
대표 질문이 버튼의 범위는 화면 변경만인가요, DB 저장까지인가요?
문서 문장예약 신청 버튼은 권한 확인, 중복 방지, DB 저장, 성공·실패 화면 반영까지 포함한다.
버튼 하나라도 DB 저장, 권한, 알림, 관리자 화면까지 붙으면 별도 기능 범위로 본다.

2부 · 채용

채용은 우리 서비스에 필요한 기능부터 정리한다

면접 장면 "채팅 기능 해봤습니다"

우리 서비스에 필요한 기능과 연결되지 않으면 판단 기준이 흔들린다.

1단계우리 서비스에 필요한 기능 정리
2단계그 기능을 확인할 면접 질문 정리
3단계좋은 답변을 찾고 이끌어낸다

1. 필요한 기능 정리

우리 서비스에 필요한 기능 경험부터 정리한다

우리 핵심예약 변경상태값 · 알림 · 취소 기준
후보 A채팅 UI매칭 약함
후보 B관리자 권한매칭 강함
후보 C알림 누락 처리매칭 강함
대표 판단먼저 우리 서비스 핵심 기능을 적어야, 후보 경험이 필요한 경험인지 판단할 수 있다.

2. 면접 질문 정리

필요한 기능을 면접 질문으로 바꾼다

01무슨 서비스?목적 · 사용자
02본인 역할?화면 · API · DB
03직접 만든 기능?입력 · 처리 · 저장
04실패하면?권한 · 중복 · 오류
05운영에서?로그 · 배포 · 복구
대표 판단후보가 말한 프로젝트를 우리에게 필요한 기능 질문으로 좁혀서 본다.
후보 답"결제 연동 해봤습니다."
추가 질문실패 화면, 중복 결제, 환불 기준, 운영 기록까지 물어본다
추가 질문 필요: 성공했다는 말만 반복함 실제 경험 신호: 실패·중복·기록까지 설명함

3. 좋은 답변 찾고 이끌기

좋은 답변은 역할과 근거가 보인다

흐린 답"팀에서 결제 연동했습니다."본인 역할이 안 보임
좋은 답"저는 실패 화면과 재시도 안내를 맡았습니다."맡은 범위가 보임
대표 판단 기준고객에게 보이는 실패 화면까지 설명하면 실제 구현 경험에 가깝다.

3. 좋은 답변 찾고 이끌기

작은 상황 질문을 하면 답변이 구체해진다

면접 질문 예시관리자 목록에 상태 필터가 필요했다면?

어떤 화면 · 데이터 · 예외를 먼저 확인할지 묻는다
